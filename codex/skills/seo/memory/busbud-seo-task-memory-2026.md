# Busbud SEO Task Memory

## Core Concept
This is the reusable Busbud SEO operating note. Use it when the task touches Busbud page-template performance, metadata systems, operator or route CTR, Bing-specific findings, or the Buson migration PRD stack. It exists to stop the team from re-discovering the same diagnosis each time.

## Stable Diagnosis To Reuse
- Busbud's main SEO opportunity is not broad crawl repair. The strongest repeatable upside is template improvement on pages that already earn impressions.
- The most important recurring surfaces are operator pages, route pages, station pages, and POI-route pages.
- Metadata is a system problem, not a copywriting problem on a few URLs.
- Bing and Google do not show the exact same failure pattern, but both point toward page selection, snippet quality, and template prioritization rather than a single catastrophic technical defect.
- Buson migration work should be treated as architecture plus crawl-control plus redirect-governance work, not just page-copy migration.

## What To Check First On Any Busbud SEO Task
1. Is the problem a ranking problem, CTR problem, crawl/indexation problem, or reporting-quality problem?
2. Which page family owns the business impact: `route`, `operator`, `station`, `POI-route`, `train-route`, `city`, or `country`?
3. Is the issue Google-facing, Bing-facing, or cross-engine?
4. Is the task about current performance optimization or Buson migration architecture?

## Highest-Value Reusable Conclusions
- `route` pages are the biggest business case by click volume.
- `station` pages are the clearest pure intent-mismatch case.
- `POI-route` pages are structurally overloaded by long entities and promo-heavy formulas.
- `train-route` pages need their own treatment and likely have both CTR and ranking weakness.
- Busbud's metadata formulas are too long, too volatile, and too dependent on price and promo language.
- Busbud already has real domain strength; the faster gains come from click efficiency, trust surfacing, and template clarity.
